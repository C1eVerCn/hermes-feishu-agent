01执行摘要

本项目（hermes-feishu-agent）是面向台架预约（test-bench reservation）业务域的飞书智能体。它把 LLM（minimax-M2.7-highspeed）作为意图解析层，把飞书 DM 作为交互界面，把后端 REST API 作为执行层，让用户能用自然语言完成真实预约。其核心矛盾是：LLM 不可信，但用户体验必须自然。围绕这一矛盾，整个系统建立在一套"双层防御 + 显式确认"的工程模式上。

04系统架构

DMZ 智能体采用四层关注点分离的设计：飞书 I/O → 业务调度 → 业务实现 → 横切控制。每层只做一件事，跨层通信通过明确定义的接口（事件、回调、工具调用）完成。横切的 OCL（Output Control Layer）贯穿所有层，承担权限、格式、身份等横切关注点。

这种分离带来的实际好处是：替换飞书为别的 IM 时只动第①层，替换 LLM 时只动第③层，新增业务域时只动第⑤层，收紧权限策略时只动 OCL。

4.1 架构总览

图 4-1：DMZ 智能体系统架构。自上而下是请求流，自下而上是响应流；OCL 作为横切层贯穿第②-⑤层。

4.2 关键设计原则

  • 关注点分离：飞书 I/O 不知道 agent 存在，agent 不知道飞书存在，工具不知道权限存在。所有跨层协议都是显式函数签名。
  • 权限不是 prompt：所有"LLM 必须先 X 再 Y"的安全策略都搬到工具注册层、装饰器层、API 表面层。prompt 只能写"建议"，不能写"必须"。
  • 状态分两类：跨请求的"长期状态"（reservation_store、identity_map、agent_pool）持久化；单次交互的"短期状态"（dry_run_state）走内存 + TTL。
  • 身份不暴露给 LLM：emailAddress 在 bench_tools 内部从 open_id 注入；LLM 的工具参数中永远见不到这一字段（VLM 域则无此字段，因为后端不鉴权）。
  • 异步优先：飞书 WS 回调必须立即返回；所有慢操作（LLM 调用、HTTP 后端、通知推送）都通过队列/线程池异步化。

05系统流程

本节描述一条用户消息从"发出"到"得到回复"的完整生命周期。下图展示了请求流与响应流的每一个节点，以及在每个节点上系统做了什么判断、读了什么状态、可能走什么分支。

阅读这张图时建议关注三件事：① LLM 不可信——所以红色防御节点不可省略；② 状态分两类——dry_run_state 是临时的、reservation_store 是持久的；③ 异步边界——所有需要等的事情都不阻塞 WS 回调线程。

5.1 端到端时序

图 5-1：单条消息的端到端流程。绿色=用户交互边界 · 红色=必经的权限防御 · 橙色=状态写入与异步通知

5.2 关键节点详解

② WS 事件入队：保证回调立即返回

飞书 WS 客户端有 3 秒超时窗口，若 3 秒内未 ack 会重发。系统采用"入队即返回"模式：ws_client 解析 P2 事件后立刻 ack，业务逻辑在另一个线程池中消费。这样无论 LLM 调多久都不会触发飞书重发。

③ 文本拦截：把"确认/取消"从意图识别中剥离

这是本项目对"LLM 不可信"的核心应对。当 dry_run_state 中已有该用户的待确认预约时，新文本"确认"或"取消"不能交给 LLM——必须走 handler 内部快路径，直接消费 dry_run_state、调真实 reserve_bench。这样 LLM 完全无法跳过 dry-run 确认步骤。

⑥+⑦ L1 + L2 双层防御：两层独立、必须都过

L1（hermes 框架钩子）和 L2（装饰器）在不同位置校验同一件事——当前用户的角色是否 ≥ TOOL_MIN_ROLE 矩阵中的最低要求。L1 失败 L2 兜底；L1 bug 触发 L2 拦截；L1 L2 都过才放行到工具 handler。这种冗余是有意为之——单点防御一旦被 prompt injection 绕过就会出大事。

⑧ emailAddress 服务端注入：LLM 永远看不见身份

台架预约工具的 schema 中没有 emailAddress 字段。Handler 在调真实后端前，从 thread-local / ocl.session_map 读出当前 user 的 email，注入到 HTTP 请求体中。LLM 看到的工具签名里完全没有这一字段——它既不能编造也不能泄露。这是"权限不出 schema"原则的具体实践。

⑨ OCL pipeline：四阶段串联处理

四个阶段按顺序应用，每阶段可独立关闭：

  • format_control：单 pass line classifier，识别 LLM 内部 turn marker（"###TURN###"）、工具调用 JSON 泄漏（"{"name": ...}"）、"建议："块等内部符号，物理删除。
  • content_filter：关键词阻断（API key 模式、敏感词），命中即替换为脱敏字符串。
  • length_limiter：在句子边界（句号/问号/换行）截断过长回复，避免单条消息过大触发飞书限制。
  • card_builder：根据响应类型（dry_run 确认卡 / 查询结果卡 / 缺字段补全卡 / 错误卡）渲染不同的飞书卡片 schema。
⑪ 异步通知：双 DM 真消息，非 toast

当真实预约成功或审批结果产生时，notify 模块从 reservation_store 反查申请人 open_id（reservation 创建时已存），并按以下规则推送：

  • 申请人 DM："你的预约已提交，调度员会尽快审批"（真消息，非 toast）
  • 调度员 DM："有新预约待审批 [链接]"（含 reservation_id，可一键审批）
  • 审批通过/拒绝：申请人 DM 通知最终结果

所有通知走 sender.send_to_user，自带 429 重试 + 分块，失败有兜底重试队列。

6.2 显式记忆层 · DMZMemoryProvider（已实现）

DMZ 智能体已实现 hermes 的 MemoryProvider 协议，跨会话持久化三类信号：用户偏好、近期操作模式、错误模式累计。这一层的价值是：LLM 在新会话开始时就能看到"这个用户常查 1.0 架构台架、最近 20 步里连续 3 次调 list_architectures 失败"等关键信号，避免重复犯错。

存储内容：

  • 用户偏好：常查的台架架构（如"1.0 架构"）、常关注的 VLM 场景名（如"hotupdate_filter_..."）
  • 近期操作模式：最近 20 步工具调用的工具名 + 参数 key 名（不存参数值，避免敏感信息）
  • 错误模式累计：4xx/5xx 错误模式；prefetch 时给 LLM 提示"用户曾频繁遇到 X 错误，注意规避"

存储位置： ~/.hermes/dmz_memory/<user_hash16>/memory.json。用户身份通过 16 位 hash 匿名化，不存真实 open_id。

不可变的安全铁律：

  • 不存 emailAddress / API key / 密码 / cookie
  • 不存完整用户消息原文（仅元数据：工具名 + 参数 key 名 + 成功/失败）
  • 不存 OCL 安全规则或权限配置
  • 30 天 TTL 自动过期
  • 匿名用户（无 open_id）不落盘
  • 落盘前 _strip_sensitive 双重保险剥敏感字段

6.3 隐式工程进化 · 从"观察 → 简化 → 巩固"

除显式 Memory 层外，系统还有一系列隐式自进化——从"观察到的痛点"演化为"代码层的固化"。这些是手动驱动的工程级进化，但每一条都对应真实可观测的失败模式：

观察到的痛点演化的产物效果
LLM 持续发明错误 key 名（task / purpose / snake_case） _KEY_ALIASES 映射表 + _normalize_args unwrap 防御 LLM 怎么写都能用，工具调用错误率从 ~30% 降到 <2%
format_control 5-pass 串行正则太慢 合并为单 pass line classifier 单次处理耗时从 ~50ms 降到 ~5ms
只有 L1 防御，被 prompt injection 绕过风险高 L1 + L2 双层防御（装饰器强制包裹） 单层 bug 仍有兜底
LLM 物理上能调 reserve_bench，可被 prompt 注入绕过 dry-run 把 dry-run 守门搬到 API 表面：reserve_bench 不注册到 LLM 可见工具集 无任何代码路径能让 LLM 跳过 dry-run
WS callback 被 time.sleep 阻塞 → 飞书 3s 超时重发 WS 回调入队即返回，业务逻辑异步消费 callback 永远 < 100ms，飞书不再重发
notify.py 与 sender.py 重复实现 client/throttle/send 删除重复代码，复用 sender.send_to_user 维护成本降低，单点修复
reservation_store 自己写 JSON 持久化 抽 infra/json_store.py 通用 helper 可复用于未来其他 state
旧 email 不刷新导致权限误判 auto_register 后调 update_profile 刷新 用户角色变更后能立即生效

这些演化的共同特征是：不依赖 LLM"按规矩写"，而是假设 LLM 一定会犯错，错了就给它兜底。每加一条 alias、每一层 decorator、每一道归一化，都是把一次"被坑过"的经验凝固进代码——这就是"工程级自进化"。

6.4 反馈闭环 · Phase 2（规划中）

Phase 2 的核心是建立"用户操作 → 反馈数据 → prompt 调整"的闭环：

  • 卡片按钮回调学习：记录用户点了哪些按钮、跳过了哪些确认，作为"提示词有效性"的信号
  • 人工反馈目录：data/feedback/ 存管理员标注的"哪类回复不友好" / "哪类 prompt 易触发误调"
  • 周报分析：自动统计失败模式（哪些工具失败率高、哪些 prompt 触发 LLM 误判）

关键约束：反馈只调整 prompt 和工具描述，不调整 OCL 安全规则。权限层永远不可被反馈改写——这是"自进化"的硬边界。

6.5 Curator 集成 · Phase 3（规划中）

Phase 3 启用 agent/curator.py，让它作为"工具 schema 巡检员"定期检查：

  • 工具描述是否仍然准确（与最近 100 次调用对比）
  • 是否有冗余工具可以合并
  • 新错误模式是否需要新 alias

关键约束：Curator 只生成 Skill / 工具描述建议，不写业务规则。所有建议进入人工 review 后才会被采纳。这是 hermes 自进化能力与"权限层不可被反馈改写"原则的边界划定。

07技术点 · Agent 权限控制

LLM 驱动的 agent 系统中，prompt 不是安全边界。所有"用户必须先 X 再 Y"、"敏感操作必须二次确认"的安全策略，都需要从 prompt 层搬到代码层。本节说明 DMZ 智能体如何用"双层防御 + API 表面隔离"实现这一目标。

7.1 双层防御（L1 + L2）

权限校验在两个独立的代码位置各做一次。两层实现不同、调用栈不同、即使一层被绕过另一层仍能拦截。

L1 — pre_tool_call 插件（硬拦截）

由 hermes 框架提供的钩子，在 LLM 决定调工具之后、工具执行之前运行。它直接修改 hermes 的回调链，进程崩溃 = 所有工具全停。

实现位于 hermes_plugins/feishu_acl：先从 ocl.session_map 取 email/role，再查 TOOL_MIN_ROLE 矩阵，不满足即抛 PermissionDenied，hermes 把它转成 LLM 可见的 tool_result 让模型自行决定后续行为。

L2 — guarded() 装饰器（兜底）

所有 handler 在 registry 注册时强制包裹 guarded(name, handler)。每次调用前从 thread-local 上下文取 user/email，再做一次角色校验。

L2 的设计目的是：L1 抛异常、被禁用、hermes 框架升级后行为变化时仍能拦截。两层独立维护、互不依赖——L1 改 L2 不动，反之亦然。

7.2 TOOL_MIN_ROLE 角色矩阵

角色 1/2/3 是粗粒度标签（普通用户/调度员/管理员），ocl/permission.py 维护工具 → 最低角色的映射。映射原则：

  • 查询类工具几乎全 1（普通用户就能查）
  • 修改类工具（reserve / cancel / approve）也是 1（粗粒度），但通过下一节的 API 表面隔离来收紧
  • 系统级工具（VLM 同步、管理员命令）固定 3

7.3 关键创新：把"dry-run 守门"搬到 API 表面

这是本项目最关键的安全决策。

问题：原本要求 LLM"必须先调 dry_run 再调 reserve"，但 prompt 不是安全边界——模型可被 prompt injection 绕过、上下文溢出后丢指令、模型降级后行为不可预测。

解法：bench_tools/register.py 只把 dry_run_reserve_bench 注册到 LLM 可见工具集，reserve_bench 仅作内部函数被 card_action_handler.confirm_reserve 回调使用——LLM 在物理上找不到能直接调真实预约的工具。

效果：即使 prompt 注入、模型降级、上下文溢出、hermes 钩子全部失效，reserve_bench 也不在 LLM 视线内。无任何代码路径能让 LLM 跳过 dry-run 卡片点确认。这是 prompt 工程师和系统架构师之间最重要的边界划定。

7.4 文本驱动确认（替代按钮）

原方案：dry-run 卡上有「✅ / ❌」按钮，用户点击 → 卡片回调处理。问题：飞书按钮在某些客户端体验不一致，且回调路径增加了一条调试链。

新方案（参考车辆预约流程）：用户打字「确认」/「取消」 → handler._handle 在交给 LLM 前先做文本拦截 → 命中确认词表则直调 reserve_bench，整个流程不经过 LLM。

这套机制的关键是：拦截发生在 agent 调度之前，LLM 既看不到确认文本、也无法拒绝拦截。配合 §6.3 的 API 表面隔离，双重锁死了"必须经过 dry-run"这一安全不变量。

08技术点 · Agent 边界控制

除权限外，agent 还需控制三件事：输出什么（不漏内部 token / 工具 JSON）、输入什么（不让恶意 prompt 穿透）、何时停（不能无限轮询）。

8.1 输出边界（OCL pipeline）

LLM 的最终回复在送给用户前必须经过四阶段处理。这四阶段在 ocl/pipeline.py:apply() 中按顺序串联，每阶段可独立关闭。

  1. format_control：单 pass line classifier 一次扫描整段文本，识别并物理删除内部 turn marker、工具调用 JSON 泄漏、"建议：" 块。早期版本是 5-pass 串行正则，已合并为单 pass 显著提速。
  2. content_filter：关键词阻断（API key 模式如 sk- 前缀、敏感词表），命中即替换为脱敏字符串并 log 一条事件。
  3. length_limiter：在句子边界（句号 / 问号 / 换行）截断过长回复，避免单条消息超过飞书大小限制。
  4. card_builder：根据响应类型（dry_run 确认卡 / 查询结果卡 / 缺字段补全卡 / 错误卡）渲染不同的飞书卡片 schema。这是 LLM 输出"长成卡片样子"的唯一通道。

8.2 prompt-level 强约束（system prompt）

不是安全边界，但 LLM 倾向服从显式命令。我们在 system prompt 中写明几条"必须"规则，配合 OCL 的硬约束形成 prompt + code 双层防线：

  • 必填字段缺失处理：缺哪个字段就问哪个，禁止瞎填默认值。
  • 用户回复是字段答案不是新意图：当用户刚被问"taskName 是什么"时，回"感知压测"必须识别为 taskName 字段值，而不是新查询。
  • list_my_reservations 默认 status=[0,1,4]：排除已取消 / 已拒绝（status=2,3），避免给用户看"假阴性"。
  • list_my_approvals 默认 status=0：只看待审批，已审批过的不要混进来。
  • 权限被拒时：明确告诉用户当前缺什么权限 + 引导联系管理员，不要尝试"绕过"。

8.3 迭代上限与超时

LLM agent 的"无限循环"问题必须用硬上限解决。本项目用三个独立机制：

参数值作用理由

8.4 数据流隔离（避免双重展示）

当 LLM 文本回复中已经枚举了所有数据时（例如"查询到 3 个台架：A、B、C"），OCL 自动抑制 data block 的附加展示——避免用户看到"文本里说 A B C，卡片里又列一遍 A B C"的双重展示。

判定方法：对 LLM 文本做标点 / 符号归一化（去掉引号、星号、连字符、括号等），然后检查所有 item 是否都能在归一化文本中找到。是 → 抑制 data block；否 → 保留 data block 作为补充。

09技术点 · LLM 行为防御层

LLM 持续发明错误 key 名、错误字段嵌套。这是结构性问题——任何 prompt 工程都只能降低频率、不能消除。唯一可靠的应对是把"容错"做在代码里，让模型怎么写都能用。

9.1 Key alias 映射（已知错名 → 正确名）

维护一张 _KEY_ALIASES 映射表，把 LLM 常写错的 key 名（task / task_name / name）映射到 schema 要求的正确名（taskName）。每个工具 handler 在解析 args 时先做一遍 alias rename，再校验必填字段。

新增工具时，alias 表是"已知 LLM 错误"的累积知识库——每发现一个错名就加一条，下次自动修复。

9.2 Unwrap 防御（包装 dict 暴力拆解）

LLM 的更高阶失败模式：把所有字段塞进一个 wrapper key（{"reservation": "{...}"}）甚至把整个对象 JSON-stringify 后作为单一字段值。

归一化层先做一次"解包装"：如果 args 是单 key 且 value 是 string，尝试 JSON-parse 拆开；拆开后是 dict 则替换 args；解析失败则原样保留（避免破坏其他情况）。

这种"暴力拆解"策略看似粗暴，但在 defense-in-depth 思路下反而最可靠——它不依赖 LLM"按规矩写"，而是假设 LLM 一定会犯错、错了就给它擦屁股。

9.3 dry-run 必填字段早检

dry-run 必须有 5 个字段（benchNo / startTime / endTime / taskName / testPurpose）。LLM 漏填时不能直接失败——必须引导用户补全。

具体做法：dry-run handler 返回 missing_fields: [...] 标识 + 正常查询结果（如果有），OCL 看到 missing_fields 后调用 card_builder.render_missing_field_card() 渲染"补字段卡"（无按钮，纯提示），并把当前已填字段写回 dry_run_state。

用户回复字段值后，下一次 LLM 调用时 system prompt 强指令："用户回复是字段答案、不是新意图" + "必须保留之前已填字段"。配合 OCL 的文本拦截（§6.4），补字段过程也不需要 LLM 主动解析语义——只要 LLM 把新字段值塞回去即可。

9.4 字段持久化与恢复

用户补字段的过程可能跨越多轮对话（"等我想一下 taskName" → 干别的 → 回来再约）。dry_run_state 把已填字段放在内存 dict + TTL，跨轮保留。TTL 过期则整个 dry-run 失效，用户必须从头开始。

这套设计的好处是：用户的"想一下"不被技术打断，但也不会无限期挂着一个待确认预约消耗系统资源。

10技术点 · 性能优化 · 速度提升

从飞书 DM 发出文字到收到回复，系统的总耗时大致分布在多个环节。本节先呈现性能地图（哪慢、占多少），再逐条说明已实施的优化、规划中的优化，以及最终目标。

本项目的核心目标是把用户感知延迟从 5-10s 降到 < 5s。关键拐点是 streaming response 上线——这一项单独就能让用户感知延迟减半，即使后端耗时不变。

10.2 已实施的优化

① iter 上限收紧（3 → 2）

把 AGENT_MAX_ITERATIONS 从 3 降到 2。绝大多数查询 1 iter 就完成，2 足以覆盖所有"调一次工具 → 看到结果 → 再调一次"的场景。iter=3 容易跑飞（LLM 自我循环调同一工具）。

效果：最坏情况下的总耗时从 ~12s 降到 ~7s。

② 硬超时收紧（120s → 30s）

AGENT_TIMEOUT_SECONDS 从 120 降到 30。30s 足够 10 轮对话，比"挂死 120s 等用户反馈"体验好得多。超时后系统返回当前最佳回复，不留半成品状态。

效果：异常情况下的"假死感"消失，资源不浪费。

③ format_control 单 pass 合并（5-pass → 1-pass）

原实现用 5 个串行正则匹配（turn marker、工具 JSON、"建议：" 块、列表项、代码块），每次匹配都遍历整段文本。优化后用一个 line classifier 一次扫描，识别所有需要删除的行。

效果：单次处理耗时从 ~50ms 降到 ~5ms。

④ WS 回调入队即返回

ws_client 收到 P2 事件后立即 ack，飞书 3s 超时窗口不会触发。业务逻辑在独立 ThreadPoolExecutor 中消费队列，5 workers 并发。

效果：callback 永远 < 100ms，飞书不再重发；用户感觉"机器人立刻收到消息"。

⑤ 通知异步化 + 429 retry + chunking

notify 模块从同步 time.sleep 改为 ThreadPoolExecutor 异步推送；复用 sender.send_to_user 的 429 指数退避重试 + 长消息分块。

效果：审批通知不再阻塞主流程；飞书限流时系统自动退避重试，用户无感知。

⑥ emailAddress 服务端注入（减 token）

台架工具的 schema 中没有 emailAddress 字段，handler 从 ocl.session_map 取 email 注入到 HTTP 请求体。LLM 不需要"想"用户身份、不需要在 prompt 里携带 email 信息、不需要在回复中泄露 email。

效果：单次工具调用的 prompt token 减少 ~50-100 个；LLM 思考更聚焦。

⑦ ThreadPoolExecutor 并发 5 个 agent

handler 的 _executor 配 5 个 worker，突发多用户消息时不串行。超出 5 的请求自动排队。

效果：低负载下即时响应；高负载下排队有序，不会"全卡死"。

10.3 规划中的优化

prompt 精简（50 行 → 25 行）

当前 system prompt ~50 行，包含角色定义、必填字段规则、确认拦截、权限提示等。计划：把可由 OCL 强约束的规则从 prompt 移到代码，prompt 只保留 LLM 真正需要"思考"的指令。

预期效果：LLM 上下文更短，首 token 延迟降低 ~20%。

system prompt 缓存

利用 LLM API 的 system prompt cache 能力（minimax 支持）。同一用户的 system prompt 在 5 分钟内不变，缓存命中可省 ~80% 的 prompt token 计费。

预期效果：单次调用成本降低、延迟降低 ~15%。

streaming response + typing 提示

当前是"等 LLM 完整生成 → 一次性发卡片"。改为 streaming：LLM 第一个 token 出来就先发 typing 提示（"机器人正在输入..."），后续 token 流式追加。

预期效果：用户感知延迟从 5-10s 降到 1-2s（实际延迟未变，但人感知更短）。

结构化错误 fallback

当前 API 5xx / 网络超时时用户看到通用错误。计划：把错误分类（后端 5xx / 网络超时 / LLM 拒答 / 权限被拒），每类对应不同提示卡。

预期效果：用户能区分"是系统问题还是我操作问题"，减少重复询问。

本阶段的核心目标是把用户感知延迟从 5-10s 降到 < 5s。关键拐点是 streaming response 上线——这一项单独就能让用户感知延迟减半，即使后端耗时不变。

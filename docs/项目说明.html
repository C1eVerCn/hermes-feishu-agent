DMZ 智能体 · 项目说明

一个基于 hermes-agent（NousResearch）+ Minimax 的飞书机器人，通过 WebSocket 接收消息， 在真实通道上探索受控 LLM 输出：格式、内容边界、权限、工具调用边界与输出长度。

整合两个业务域：台架预约（REST API @9013）与 VLM 精标数据（dmz-ess-vlm REST API @9014）。

Python 3.11+ hermes-agent lark-oapi (WebSocket) FastAPI · httpx 358 单元测试全绿

1项目整体架构

分层清晰、一层一职责；框架代码与业务域解耦，换业务域只动 *_tools/ 与配置。

1.1 端到端数据流

2项目工作流程（台架预约）

分层拦截，把高频确定性操作从 30s 的 LLM 路径上摘下来，降到 <1s；只有自由形态请求才进 LLM。

2.1 消息进入后的判定顺序

  1. 身份解析（最早）：open_id → 飞书 Contact API 取 email/name → auto_register 建档 → 解析 role（含 OCL_ADMIN_USER_IDS 自动升 role 3）。
  2. Layer 0 简单意图：你好 / 帮助 / 谢谢 等正则命中 → 即时卡片回复，不进 Agent。
  3. Layer 0.5 查询快速路径：「查可用台架」「我的预约」「台架架构」等 → 直接调 bench_handlers（已被 guarded() 包裹）→ OCL 卡片，<1s。
  4. Layer 0.6 预约快速路径：「预约 TJ001，从明天下午5点到后天晚上8点，任务是X，目的是Y」→ 正则抽参 + 中文时间解析 → dry_run_reserve_bench → 确认卡片。
  5. 身份 / 管理命令：我的权限 设置角色 ou_xxx 2 查看用户 → bypass Agent，确定性处理。
  6. 身份闸：role==0（非平台用户）→ 友好提示联系管理员开通。
  7. 确认/取消拦截器：存在待确认 dry_run 且用户回「确认/取消」→ 确定性执行真实预约（不经 LLM）。
  8. Agent 调用：以上都不命中 → 注入「已核验身份」前缀 + 原文 → agent.chat() → OCL → 卡片。

2.2 预约两步流程（dry_run → 确认）

用户：预约 TJ001，从明天下午5点到后天晚上8点，任务是标定，目的是感知压测 │ ▼ Layer 0.6 正则抽参 + 中文时间解析（UTC+8） dry_run_reserve_bench(args) ← 真实 reserve_bench 对 LLM 不可见 │ 返回确认卡片（含全部字段，缺字段则提示补全） ▼ 保存 dry_run_state[user_id] │ ▼ 用户回「确认」 确认/取消拦截器 → 权限闸(reserve_bench) → set_current_email → reserve_bench(真实) │ ├─ 通知本组调度员（best-effort，DM） ├─ 申请人确认卡片 └─ reservation_store 落盘 (benchNo,startTime → applicant_open_id)，供审批通过后回 DM 申请人
为什么 dry_run 在前：弱模型容易传错参；先用确定性确认卡片把全部字段摆给用户核对， 点「确认」才真正下单，并触发调度员通知——把易错环节从 LLM 手里拿走。

2.3 卡片按钮真回调（确定性，不经 LLM）

查询/预约卡片上的按钮点击是同步回调，由 bot/card_action_handler.py 处理： 取点击者 open_id → 注入 email + OCL 角色门控 → 直接调对应工具 → 返回 toast。

卡片来源按钮动作 → 工具
我的预约（待审批）取消预约cancel → cancel_reservation
我的审批（待审批）批准 / 拒绝approve → approve_reservation
已批准的预约归还台架return → return_bench（缺地点则引导补充）
伪造防护：按钮 value 永不含 email——由点击者 open_id 反查，外加 OCL 角色门控 + 后端业务校验两道闸，伪造 value 也越不了权。

3权限控制设计与实现

三档角色 + 双层独立防御 + 服务端身份注入；LLM 只被「告知」身份，从不被信任来决定权限。

3.1 角色模型

角色能力台架工具（最低角色）VLM 工具
1 普通用户查询 / 预约 / 取消 / 归还自己范围内台架；查 VLM 公开数据list_architectures, list_available_benches, reserve_bench, cancel_reservation, return_bench, list_my_reservationslist_*/get_*/playback/download_metadata/frame_image_url
2 调度员+ 审批本组预约、查本组待审批approve_reservation, list_my_approvals+ 数据导出下载
3 管理员+ 跨组审批、触发 VLM 同步等系统级操作（全部）sync_execute, trigger_sync_async, sync_status

3.2 身份从哪来（open_id / email 作为识别码）

每条飞书消息都带发送者 open_id。身份解析链：

  • open_id → 飞书 Contact API 解析 email / name（内存缓存 + 负向缓存，单次调用 2s 超时防抖）。
  • role 来源（优先级）：data/identity_map.json 的管理员覆盖 → OCL_ADMIN_USER_IDS 环境变量（自动升至 role 3 并持久化）→ 飞书能拿到 email 则默认 role 1 → 否则 role 0（非平台用户）。
  • 身份记录由 bot/identity_admin.py（v2 schema）持久化到 data/identity_map.json，所有写操作落审计 data/identity_audit.jsonl。
近期修复①（LLM 角色失认）：过去 Agent 只收到用户原文，身份对 LLM 不可见，导致管理员被 LLM 说成「普通用户 role=1」。 现已在每轮注入服务端已核验的身份前缀（角色 + 权限范围，不含 PII），见 handler._identity_preamble()。权限执行本就独立于 LLM，此修复纠正的是 LLM 的表述。
近期修复②（两套 admin 不一致）：OCL_ADMIN_USER_IDS 过去只对「管理命令」生效，不影响 role 变量。 现统一为 _resolve_role_with_env_admin()：env 内的管理员在所有路径都解析为 role 3，并写回 store 作为唯一事实源。

3.3 双层防御（两道独立的闸）

Layer 1 · pre_tool_call 插件（硬拦截）

位于 hermes 内部、工具执行前最后检查点。session_id → session_map.lookup() → user_id → is_tool_permitted()。

返回 {"action":"block"} 阻断；任何异常 fail-open（放行），由 Layer 2 兜底。

Layer 2 · guarded() 包装器（兜底）

位于工具 handler 入口。通过 threading.local 读 get_current_user() → is_tool_permitted()。

覆盖 Layer 1 失效场景（插件未加载 / config 误删 / session_map 缺失）。

OCL 只做粗粒度门控（按 TOOL_MIN_ROLE 表能否调某工具）；细粒度规则（同组、状态机限制）由后端按 emailAddress 强制——两道相互独立的闸，演示分层防御。

3.4 emailAddress 服务端注入

台架域工具 schema 不含 emailAddress；调用前 handler 用 set_current_email(email) 写入 thread-local， bench_handlers 发 HTTP 时从上下文注入。弱模型既无法编造邮箱，也无法给别人预约。VLM 域则完全不注入身份。

4输出边界控制设计与实现（OCL）

弱模型输出不可信：可能泄漏 tool-call JSON、内部 turn 标记、超长或越界内容。OCL 在发送前做确定性兜底。

4.1 流水线

ocl/pipeline.apply(response, user_id, captured) │ ├─ format_control 剥离 tool-call JSON / "新消息"/"System:" 等内部 turn 标记；空响应 → 兜底语 ├─ content_filter 正则检查（政治敏感、密钥泄漏）→ 命中则整条替换为 OCL_CONTENT_BLOCK_MESSAGE ├─ length_limiter 硬截断（默认 4000 字），保护飞书消息上限 └─ card_builder 文本(摘要) + 捕获的工具原始 JSON → 互动卡片 ▼ OclResult(text, blocked, block_reason, card)

4.2 确定性卡片渲染

通过 hermes post_tool_call 钩子把每个工具返回的原始 JSON 按 session 存入 ocl/tool_capture.py； card_builder 用原始数据渲染卡片（无标题栏：摘要块 + 确定性数据块 + 交互按钮），数据不经弱模型转述。

  • markdown_to_lark.py：# 标题 → 加粗，去掉飞书不支持的 ``` 代码围栏，解决字面星号/横线问题。
  • 状态以文字徽标体现：🟡待审批 / 🟢已批准 / 🔴已拒绝 / ⚪已取消 / ✅已完成。
  • 只对参数自足的动作放真回调按钮；可用台架的「预约」缺时间/任务 → 不放按钮，走对话。

4.3 系统提示里的硬输出边界

构造 Agent 时通过 ephemeral_system_prompt 注入硬规则（节选）：

  • 禁止在正文输出 tool-call JSON、System:/Human:/--- 等内部标记。
  • 工具返回的具体值（台架号/架构/时间/BagId）由卡片单独呈现，正文只写一句总结 + 下一步提示。
  • 单轮最多 2 次工具调用；台架号 vs 架构名格式区分；dry_run 字段按用户原话字面填，不语义重写。

4.4 不变量

约束说明
<100msOCL 全程无网络调用（card_builder 纯 CPU）
不抛异常任何处理错误 fail-open，返回原文，不阻断发送

5Hermes 自进化：约束与方向控制

让系统跨会话「记住」与「自检」，但方向被严格收束：只允许往「更懂用户、更早预警、产出只读建议」走，绝不允许触碰安全/权限/业务规则。

5.1 三阶段现状

阶段模块状态作用
Phase 1 跨会话记忆bot/dmz_memory.py已接线实现 hermes MemoryProvider，每用户记忆常查架构/场景、最近工具调用模式、累计错误模式
Phase 2 反馈回路bot/feedback.py已接线卡片按钮回调记录「操作模式」（仅元数据）落盘 data/feedback/operations，供周报分析
Phase 3 Curatorbot/curator_runner.py已接线 · 只读后台巡检工具 schema / Skill / 失败模式 → 仅产出建议到 data/curator/suggestions

5.2 Phase 1 记忆：接线方式

hermes 只从 plugins/memory/ 发现 provider；我们的 provider 在 repo 内，故不走插件发现，而在 agent_pool.get_or_create() 构造 AIAgent 后直接挂载：

AIAgent(..., user_id=user_id)        # 硬前提：不传 user_id 则记忆永不落盘
  └─ _wire_dmz_memory(agent, session_id, user_id)
        MemoryManager → add_provider(DMZMemoryProvider()) → initialize(...)
hermes 运行时自动：每轮 prefetch（召回）+ 每轮末 sync_turn（写回）

存储：$DMZ_MEMORY_HOME/dmz_memory/<user_hash16>/memory.json（默认 data/，挂载进容器卷，跨重启持久化）。

5.3 安全铁律（不可变）

记忆 / 反馈层绝不存：❌ emailAddress / API key / 密码 / cookie ❌ 完整用户消息原文（仅元数据：工具名 + 参数 key 名 + 成功/失败）❌ OCL 安全规则或权限配置。
必做：✅ 30 天 TTL 自动过期 ✅ 匿名用户（无 open_id）不落盘 ✅ 落盘前 _strip_sensitive 双重剥敏。

5.4 自进化「方向控制」：Curator 只读护栏

Phase 3 的核心是限制自进化能改什么。Curator 是只读 wrapper：

  • 只产出建议，写到 data/curator/suggestions/<日期>.jsonl（带 hash + status:pending），从不自动改任何代码。
  • 巡检范围：工具 schema（空描述 / schema 泄漏 email）、Skill（过期 / 缺红旗段）、反馈失败模式（高频错误 ≥3 次）。
  • BLOCKED_PATHS 显式禁改：dmz_memory.py、feedback.py、curator_runner.py、ocl/permission.py、ocl/format_control.py、ocl/content_filter.py、ocl/length_limiter.py、config/settings.py——即安全/权限/输出边界/配置永不进入自进化回路。
  • 触发方式：搭在 agent-warmup 后台线程上，内部限流 24h/次，失败静默，绝不影响服务。
设计意图：自进化只能让系统「更好用」（更准的工具描述、更早的失败预警），不能让它「自我授权」或「放松边界」。人始终在建议→采纳的回路中。

6后续开发方向

6.1 体验与性能

  • 流式 UX 重做：此前 streaming 占位（typing → 逐字 edit_message）已实现后按需求回退，保留了启动预热。可在确认 UX 后重新接入，把感知延迟从 30s+ 降到 ~2s。
  • VLM 域卡片化：当前 VLM 走纯文本/通用卡片，可仿台架域做 Bag/帧的确定性数据块与跳转按钮。
  • 快速路径覆盖更多查询意图（当前以台架查询为主）。

6.2 自进化闭环（Phase 2 / 3 深化）

  • 周报分析：定期跑 feedback.weekly_report()，把高频失败模式转成 prompt / 工具描述改进项。
  • Curator 建议审阅闭环：给 data/curator/suggestions 加人工「采纳 / 驳回」流程（仍由人决定，护栏不变）。
  • 把 Phase 3 升级为读 agent/curator.py，仅生成 Skill / 工具描述建议。

6.3 健壮性与运维

  • 内容过滤增强：当前仅政治敏感 + 密钥两类正则，覆盖面窄；生产前需扩充规则集。
  • session_map TTL：异常销毁的 Agent 映射目前依赖 LRH 上限兜底，可加主动 TTL 清理。
  • 告警接入：把 /health 指标（重连率、错误率、p95 延迟）接入监控。
  • 文档对齐：旧 README/architecture/deployment 仍残留早期「订单域」描述，需同步到本说明的双业务域模型。
不变量提醒（后续开发须守）：WS 回调立即返回；每 user_id 严格一个 Agent 实例；MINIMAX_API_KEY 不入日志； max_iterations=30 / timeout=120s 为硬上限；台架域注入 email、VLM 域不注入；记忆层不存敏感字段。
